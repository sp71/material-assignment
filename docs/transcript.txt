  Step 2 — In-memory data structure

  The five core ops are cd, pwd, ls, mv, find. Here's how each structure handles
  them, with the extensions in mind since that's where the real differences
  show up.

  Pointer tree — each directory holds map[string]*Node of children plus a parent
  *Node; files hold contents. Simplest model that's also correct. Every core op
  is at its natural complexity. Move-while-reading (streaming extension) works
  because a reader holds the *File directly, so a rename just re-keys it in the
  parent — the reader never notices. Weak spot: hardlinks. A hardlinked node has
  multiple parents, which makes the parent pointer ambiguous and pwd
  ill-defined (this is exactly why Unix forbids hardlinked directories). Doable,
  but it's bolted on.

  Flat map (full-path key) — pwd is free and absolute-path operations become
  trivial lookups, which is seductive. But it's disqualified by the core ops: ls
  needs an O(n) scan unless you maintain a per-directory child index (at which
  point you've re-implemented the tree), and directory mv rewrites the path key 
  of every descendant — O(subtree) string churn that also invalidates any held
  reference. Since mv is core and move/copy is an extension, this is a
  dealbreaker. It also can't represent hardlinks (path = identity is baked in)
  or survive move-while-reading.

  Inode table (Unix model) — separate identity (inode: contents, mode, owner,
  link count) from naming (directory = map[name]inodeID dirents). Slightly more
  indirection (two lookups) and pwd takes a bit more bookkeeping. But it's the
  model the harder extensions were designed for: hardlinks are native (N dirents
  → one inode + refcount), permissions/owner are just inode fields, and
  move-while-reading is automatic because handles bind to the inode, not the
  dirent. It's heavier than the base spec needs, and pays off only if you go
  deep on links or permissions/multi-user.

  Recommendation: pointer tree, with one forward-looking tweak

  Make the core a pointer tree (directory: children map[string]*Node + parent;
  file: contents). It's the simplest structure that gives every core op its
  natural complexity and cleanly supports the streaming/concurrency extension
  you're leaning toward — for files, move-while-reading falls out for free,
  identical to what the inode model would give you.

  The one tweak I'd make up front: have a file node hold a pointer to a separate
  fileContent struct (bytes + RWMutex) rather than embedding the bytes inline.
  That costs nothing now, keeps locking scoped per-file (good for the
  1-writer/N-reader requirement), and means file hardlinks later become "two
  dirents → one fileContent" without migrating to a full inode table.

  What would change my recommendation:
  - You pick hardlinks (especially of directories) or full Unix 
  permissions/multi-user as the deep extension → go straight to the inode table;
  retrofitting it onto a parent-pointer tree is uglier than starting there.
  - Absolute-path ops dominate and the tree is shallow & rarely mutated → flat
  map's free pwd/lookups get tempting — but the moment directory mv matters,
  it's off the table.

  Note the data structure is orthogonal to where CWD lives — CWD is just a *Node
  (or inode id) held by whoever owns the session, so this choice doesn't
  constrain the concurrency design.

  Your call on the structure — or tell me if the extension you're eyeing pushes
  us toward the inode table and we can decide them together.

⏺ Both good questions. Let me be concrete.

  1. Flat path-keyed map: what mv of a directory does

  Say the map is keyed by absolute path:

  "/school"                    -> &dir{}
  "/school/homework"           -> &dir{}
  "/school/homework/math"      -> &dir{}
  "/school/homework/math/hw1"  -> &file{bytes}
  "/school/homework/history"   -> &dir{}

  Now mv /school/homework /school/archive (rename a directory). The key
  /school/homework is just a string — renaming it is not one map operation,
  because every descendant's key embeds the parent's path. You have to:

  1. Scan to find the affected set. Find every key with prefix
  "/school/homework/". That's an O(n) scan of the entire map (Go maps have no
  prefix/range query — that's a sorted structure like a trie or B-tree, which
  you don't have), or O(subtree) if you maintain a child index — but a child
  index means you've rebuilt the tree on the side.
  2. Rewrite each key. For all k matches, delete the old key and insert a new
  one with the prefix swapped to /school/archive/.... Each is an alloc + hash +
  insert. For a subtree of k entries that's k delete+insert pairs and k new
  strings.
  3. Do it transactionally. If you mutate the map mid-iteration you'll corrupt
  your scan, so you buffer the rewrites then apply — extra memory proportional
  to the subtree.

  So a single directory rename is O(n) to find + O(k) to rewrite, versus O(1)
  re-key in a pointer tree. Under the "moderately high volume" criterion this is
  the thing a reviewer pounces on: an operation that's O(1) in the right
  structure becomes a full-filesystem scan.

  What it does to an open stream handle

  This is the deeper problem, and it's not just slow — it's semantically broken.
  Two ways you could have built the handle, both bad:

  - Handle stores a path string (handle{path: "/school/homework/math/hw1", 
  offset: N}). After the directory rename, that key no longer exists in the map
  — the next read/write does a lookup and gets "not found." The stream dies on a
  move. That directly violates the streaming requirement ("continue
  reading/writing even if it gets moved"). You could try to patch up every open
  handle's path during the rename, but now mv has to know about all live handles
  and rewrite their paths too — coupling the mutation path to the I/O layer.
  - Handle stores the *file pointer (handle{f: *file, offset: N}). Now the read
  survives the move — but you've just admitted that the pointer, not the path, 
  is the file's true identity. Which means the path-keyed map is no longer your
  source of truth; it's a secondary index over pointer identity. At that point
  the flat map is fighting you: identity lives in pointers, naming lives in the
  map, and you have to keep them consistent by hand. You've built a worse
  version of the inode model by accident.

  That second point is the real argument: the streaming "survives a move" 
  requirement is fundamentally incompatible with "path is identity," which is
  the flat map's whole premise. So the flat map isn't just slow on directory mv
  — it structurally can't give you the streaming extension cleanly.

  2. Inode table vs node tree, without hardlinks

  Honest answer: if you're not doing hardlinks, the inode table buys you very 
  little, and the node tree is the better choice. Let me separate what's real
  from what's marketing.

  What the inode layer gives you is separation of identity (inode) from naming 
  (dirent). The features that cash in on that separation are:

  - Hardlinks — N names → one inode. (You're excluding this.)
  - mv / move-while-reading — the thing being moved keeps its identity while its
  name/location changes.
  - Permissions/owner as intrinsic properties — they live on the identity, not
  the name.

  Here's the catch: in a pointer tree, the *Node pointer already gives you 
  identity separate from name. The name is the key in the parent's children map;
  the identity is the pointer. So:

  - Move-while-reading? Already handled — the handle holds *file, and mv just
  re-keys the parent map. You get this for free without an inode table.
  - Permissions/owner? Just fields on the Node struct. No inode layer needed.
  - mv? Re-key in parent map, O(1). Same as the inode model.

  So the only thing the inode table does that a pointer tree doesn't already do 
  is let one identity have multiple names — which is hardlinks. Strip hardlinks
  out and you've removed the inode table's entire reason to exist. What you'd be
  left with is pure cost:

  - An extra indirection on every traversal (name → id, then id → inode).
  - An inode-ID allocator, plus recycling/reuse logic.
  - A second map (the inode table) to keep consistent with the dirents — more
  invariants, more code, more to get wrong in review.
  - A more bookkeeping-heavy pwd (dirents don't inherently know their parent;
  you track it separately).

  That's real complexity with no payoff if hardlinks are off the table.

  Bottom line: the inode table is worth it exactly when you commit to hardlinks
  (or you want textbook-faithful Unix semantics as a talking point). For
  everything else — including the streaming/concurrency extension you're eyeing,
  permissions, and absolute/relative paths — the pointer tree gives you
  identity-vs-name separation implicitly through pointers, at lower complexity.
  That's why my recommendation stands at pointer tree unless hardlinks become
  the deep extension.

  So the structure decision is really gated on the extension. Want to settle
  Step 3 (extension) next so the data-structure choice falls out of it cleanly,
  or lock in the pointer tree now on the assumption hardlinks aren't the target?

     In-Memory Filesystem — Implementation Plan

     Context

     Material take-home: build an in-memory filesystem in Go that exposes
     functions
     equivalent to familiar shell commands (cd, pwd, mkdir, ls, rmdir,
     touch, read/write/move a file, find). No real disk I/O — the whole tree
     lives in memory. The bar is production-quality, reviewable code: clear
     naming,
     explicit error handling, and data structures that stay efficient under
     "moderately high volume."

     Decisions already made with the reviewer (recorded so we can defend them):

     - Language: Go. Explicit error returns force deliberate edge-case
     handling; sync.RWMutex/goroutines make the streaming extension a real
     showcase; net/http (no framework) keeps a future UI cheap. Verbosity is an
     acceptable cost for a readability-graded review.
     - Data structure: pointer tree (directory nodes with a children map + a
     parent pointer). Every core op lands at its natural complexity; a flat
     path-keyed map was rejected because a directory mv becomes an O(n) key
     rewrite and can't survive a move on an open handle, and a full inode table
     buys nothing over pointers unless we do hardlinks (we are not).
     - Borrowed from the inode model: identity ≠ name. A file's directory entry
     points to a shared *fileContent (bytes + its own lock). An open stream
     handle holds that *fileContent directly, so it survives rename and
     removal — Unix unlink-while-open semantics, for free via the GC.
     - Built extensions: streaming I/O (deep) + subtree walk (light).
     - Walk is deliberately minimal: a visitor callback plus a SkipDir signal.
     Core find and a regex find are layered on top of it.

     Open defaults (called out — easy to change)

     - Single, global CWD on the FS. Matches the base spec literally and keeps
     the core API simple. Per-session/per-user CWD is listed as a future
     extension. The tree is still made safe for concurrent use (see locking).
     - Delivery surface: library + tests, plus a runnable Example test that
     mirrors the spec's walkthrough (doubles as godoc). No CLI/HTTP — the
     browser
     explorer is future-only.
     - Removal split three ways, mirroring Unix: Remove(name) unlinks a single
     file; RemoveDir(name) removes a directory only when empty
     (ErrDirNotEmpty otherwise); RemoveAll(name) removes a directory and its
     whole subtree. Recursive removal pairs nicely with the streaming story:
     unlinked files with open handles stay readable until the handle closes.

     Data model

     Package memfs. All node types are unexported; the public surface is
     methods on *FS plus the stream handles, WalkFunc/SkipDir, and sentinel
     errors.

     // node is implemented by *directory and *file.
     type node interface {
         name() string
         setName(string)
         parent() *directory
         setParent(*directory)
     }

     type directory struct {
         nodeName string
         par      *directory
         children map[string]node   // name -> file | directory
     }

     type file struct {
         nodeName string
         par      *directory
         content  *fileContent      // shared identity — handles bind here
     }

     type fileContent struct {
         mu         sync.RWMutex     // guards data + writerOpen for one file
         data       []byte
         writerOpen bool             // at most one active streaming writer
     }

     type FS struct {
         mu   sync.RWMutex           // guards tree structure + cwd
         root *directory
         cwd  *directory
     }

     Pwd walks par pointers to the root, collects names, reverses, and joins
     with
     / (root renders as "/"). Identity living in the pointer (not the path) is
     what makes mv O(1) and lets handles outlive a move.

     Concurrency model

     Two independent lock levels, never held simultaneously:

     1. FS.mu (RWMutex) — guards the tree shape (children maps, parent
     pointers, cwd). Structural ops (Mkdir, RemoveDir, Move, Cd) take
     the write lock; reads (Ls, Pwd, resolution) take the read lock.
     2. fileContent.mu (RWMutex, one per file) — guards that file's bytes and
     its
     single-writer flag.

     Invariant: resolve-then-release. Any content operation resolves the name to
     its *fileContent under FS.mu, releases FS.mu, and only then takes the
     content lock. Because we never nest the two, there is no lock-ordering
     deadlock,
     and — critically — a concurrent Move (which takes FS.mu) does not block
     an in-flight Read. That is the mechanism behind "keep reading after a
     move."

     1 writer / N readers: fileContent.mu allows concurrent reads; the
     writerOpen flag (set under the lock at OpenWriter, cleared on Close)
     enforces a single streaming writer. Individual Write calls take the write
     lock
     only for the duration of the byte-slice mutation, so readers aren't blocked
     for
     the writer's whole lifetime.

     Ls/Find return copied slices so callers hold no references into locked
     state. Entire suite runs under go test -race.

     Public API & error cases

     Sentinel errors in errors.go, wrapped with context via fmt.Errorf("%s: %w",
     name, ErrX) and comparable with errors.Is:

     ErrNotFound, ErrExists, ErrNotDir, ErrIsDir, ErrDirNotEmpty,
     ErrInvalidName, ErrWriterBusy, ErrInvalidSeek, ErrClosed

     validateName rejects empty names, names containing /, and the reserved .
     and ...

     Walk (light extension) — walk.go

     type WalkFunc func(path string, isDir bool) error
     var SkipDir = errors.New("skip directory subtree")

     func (fs *FS) Walk(fn WalkFunc) error          // visits every descendant
     of cwd
     func (fs *FS) FindRegex(pat string) ([]string, error)  // bad pattern ->
     error

     Walk visits each descendant of the cwd (absolute path, isDir flag).
     Returning
     SkipDir from a directory prunes its subtree; any other error aborts and
     propagates. Kept to exactly this contract — no extra knobs. Find and
     FindRegex are thin loops over Walk.

     Streaming (deep extension) — stream.go

     func (fs *FS) Open(name string) (*FileReader, error)        // N allowed
     func (fs *FS) OpenWriter(name string) (*FileWriter, error)  //
     ErrWriterBusy if one is open

     // *FileReader: io.Reader, io.ReaderAt, io.Seeker, Close()
     // *FileWriter: io.Writer, io.WriterAt, io.Seeker, Close()  (Close clears 
     writerOpen)

     Each handle holds a *fileContent + its own offset + a closed flag.
     Open/OpenWriter errors: ErrNotFound, ErrIsDir, and OpenWriter adds
     ErrWriterBusy. After Close, any Read/Write/Seek/ReadAt/WriteAt

     validate offsets (ErrInvalidSeek); reads past end return io.EOF;
     WriteAt/Write past end extend the buffer. Random access satisfies the "jump
     to any part" requirement; handles ignore renames/removals by design.
